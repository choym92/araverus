# News Threading System Design

## Overview

Story threads dynamically group related news articles about the same developing story. Threads have no member limit and manage themselves through heat-based lifecycle with no manual intervention.

**v5.0**: Heuristic matching (20+ constants) replaced with LLM Judge (7 constants). Thread analysis and parent grouping added.

## How It Works (Plain English)

### 1. Turn articles into numbers (embedding)
Each article's title + description + summary (from LLM analysis) is converted into a list of 768 numbers (a "vector"). Articles about similar topics produce similar number patterns.

### 2. LLM Judge assigns articles to threads
When a new article arrives, the pipeline:
1. **Cosine pre-filter**: Compares the article's embedding against all active thread centroids. Threads with cosine similarity ≥ 0.40 become candidates (top 5 kept).
2. **LLM Judge**: Sends the article + candidate threads to Gemini 2.5 Flash. The LLM decides:
   - **"assign"** + thread_id — article belongs to this thread (direct or causal match)
   - **"new_thread"** — article doesn't belong to any candidate
3. **Judgment saved**: Every decision is recorded in `wsj_thread_judgments` with reason, confidence, match_type.

**Why LLM Judge over heuristics?**
- Heuristics (cosine + entity overlap + time penalty + author boost + margin check) used 20+ constants and couldn't capture causal relationships ("Fed Rate Cut" → "Housing Slump")
- LLM Judge uses 7 constants and understands context, causation, and story evolution
- Contamination drops: Restaurant Q4 thread no longer absorbs unrelated restaurant articles

**Example:**
```
Article: "Housing Prices Drop After Fed's Rate Hold"
Candidate threads:
  1. "Fed Rate Decisions" (cosine: 0.72)
  2. "Housing Market Slowdown" (cosine: 0.68)
  3. "Mortgage Rate Trends" (cosine: 0.55)

LLM Judge decides:
  → assign to "Housing Market Slowdown" (direct match)
  → related_thread_id: "Fed Rate Decisions" (causal link captured)
```

### 3. Group leftovers with AI
Articles that didn't match any thread go to Gemini for semantic grouping (same as before). If the new group's centroid is >0.92 similar to an existing thread, it merges instead of creating a duplicate.

### 4. Analyze threads (NEW — Step B)
Threads with 3+ new articles since last analysis get a full LLM analysis:
- **Updated title** (if story evolved)
- **Summary, catalyst, drivers**
- **Impacts** (stocks, sectors, commodities affected, with direction and confidence)
- **Narrative strength/velocity** (1-10 scale, accelerating/stable/decelerating)
- **Dominant theme/sector/macro** classification

Stored in `analysis_json` JSONB on `wsj_story_threads`. History snapshots in `wsj_thread_analysis_history`.

### 5. Group threads into parents (NEW — Step C)
Active threads are grouped into parent macro-events via LLM:
- "Iran Crisis" → [US-Iran Military, Gas Prices, Iran Protests]
- "Fed Policy" → [Rate Decision, Dollar Fluctuations]

Parent groups stored in `wsj_parent_threads` table, linked via `parent_id` FK.

### 6. Retire old threads
Threads with no new articles for 14+ days are archived. Same lifecycle as before.

## Thread Lifecycle

```
Active → Cooling → Archived → (Resurrected → Active)
```

| State | Condition | Matching | Frontend |
|-------|-----------|----------|----------|
| **Active** | New article within 3 days | Yes | Shown, ranked by heat |
| **Cooling** | 3-14 days since last article | Yes | Shown below active |
| **Archived** | 14+ days inactive | No matching | Hidden from feed |
| **Resurrected** | Archived thread gets a new match | Becomes Active again | Re-appears |

## Heat Score

Computed at query time (not stored). Determines frontend ranking.

```
heat = Σ (importance_weight × time_decay(article_age))

importance_weight: must_read=3, worth_reading=2, optional=1
time_decay: e^(-0.3 × days_old)  → half-life ~2.3 days
```

## Daily Pipeline Flow

```
[1/5] Embed articles (BAAI/bge-base-en-v1.5, title+desc+summary)
[2/5] Assign threads (LLM Judge: cosine pre-filter → Gemini decision)
        → Unmatched → LLM grouping → merge check (>0.92)
[3/5] Update thread statuses (active/cooling/archived)
[4/5] Analyze threads (impacts, narrative, drivers — threads with 3+ new articles)
[5/5] Group into parents (macro-event grouping — 1x daily)
```

### CLI flags
- `--legacy`: Use old heuristic matching (simple cosine threshold 0.60, no Judge)
- `--days N`: Only process articles from last N days (default 3, 0=all)
- `--rejudge`: Re-evaluate already-judged articles (default: skip them)
- `--skip-analysis`: Skip thread analysis (step 4)
- `--skip-parents`: Skip parent grouping (step 5)
- `--embed-only`: Only embed, skip thread matching
- `--dry-run`: No DB writes
- `--seed-golden <path>`: Seed from golden dataset

## LLM Judge Details

### Prompt structure
The Judge receives:
- **Article**: title, date, author, summary/description, keywords
- **Candidates**: thread title, member count, cosine score, 3 most recent article titles

### Decision types
| Decision | match_type | When |
|----------|------------|------|
| assign | direct | Article reports on same event as thread |
| assign | causal | Article about different event caused by thread's story |
| new_thread | none | Article doesn't belong to any candidate |
| no_match | none | No candidates above cosine threshold |

### Judgment audit trail
Every decision saved to `wsj_thread_judgments`:
- `candidate_threads_json`: which threads were considered
- `decision`, `chosen_thread_id`, `decision_reason`
- `confidence` (high/medium/low), `match_type` (direct/causal/none)
- `related_thread_id`: for causal links
- `judge_model`, `prompt_version`

## Thread Analysis (Step B)

### Trigger
Thread is re-analyzed when `member_count - analysis_article_count >= 3`.

### Input data (per article)
| Data | Source | Coverage |
|------|--------|----------|
| Title | wsj_items.title | 100% |
| Summary | wsj_llm_analysis.summary | 73% |
| Sentiment | wsj_llm_analysis.sentiment | 87% |
| Key entities | wsj_llm_analysis.key_entities | 85% |
| Geographic region | wsj_llm_analysis.geographic_region | 87% |
| Date | wsj_items.published_at | 100% |

### Output (analysis_json)
```json
{
  "updated_title": "US-Iran Military Escalation and Retaliation Risk",
  "summary": "미국의 시리아 공습 이후...",

```

## Parent Grouping (Step C)

Runs once daily. Groups active/cooling threads into parent macro-events via LLM.
- Input: all active/cooling thread titles + member counts (max 80)
- Output: parent groups with 2+ threads each
- Stored in `wsj_parent_threads` table, linked via `wsj_story_threads.parent_id`
- Re-computed each run (clears existing assignments first)
