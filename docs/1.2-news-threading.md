# News Threading System Design

## Overview

Story threads dynamically group related news articles about the same developing story. Threads have no member limit and manage themselves through heat-based lifecycle with no manual intervention.

## How It Works (Plain English)

### 1. Turn articles into numbers (embedding)
Each article's title + description + summary (from LLM analysis) is converted into a list of 768 numbers (a "vector"). The summary adds deeper semantic meaning from the crawled article content. Articles about similar topics produce similar number patterns — like GPS coordinates where "gas price" articles cluster near each other in 768-dimensional space.

### 2. Try to match new articles to existing threads
Each thread stores a **centroid** — a single 768-dim vector that represents the "average" of all its member articles. Think of it as the thread's GPS coordinate in meaning-space.

When a new article arrives, the pipeline:
1. **Compares** the article's embedding against every Active and Cooling thread's centroid using cosine similarity (0.0 = unrelated, 1.0 = identical).
2. **Calculates a dynamic threshold** for each thread — the minimum similarity needed to join. This threshold is NOT a fixed number; it changes per thread based on:
   - **Base (0.73)**: The starting bar. Below this, articles are too different regardless of thread state.
   - **Time penalty (+0.01 per day)**: How many days since the thread's last article. A thread quiet for 5 days needs 0.78 instead of 0.73. Prevents old dead stories from grabbing new unrelated articles.
   - **Size penalty (+0.04 × ln(members+1))**: Bigger threads are harder to join. A 2-member thread needs 0.774; a 50-member thread needs 0.87. Without this, popular threads would attract everything like a black hole — "U.S. Economy" would absorb every finance article.
3. **Picks the best match** — the thread with the highest cosine similarity that passes the threshold.
4. **Margin check**: The best thread must clearly beat the runner-up by at least 0.03. This prevents coin-flip assignments.

**Example walkthrough:**

A new article "Fed Signals Rate Pause in June" arrives. The pipeline computes:
| Thread | Centroid similarity | Threshold (base + penalties) | Pass? |
|--------|-------------------|------------------------------|-------|
| "Fed Rate Decisions" (8 members, last article today) | 0.82 | 0.73 + 0.00 + 0.088 = **0.818** | 0.82 ≥ 0.818 ✓ |
| "U.S. Inflation Data" (25 members, last article 2 days ago) | 0.76 | 0.73 + 0.02 + 0.131 = **0.881** | 0.76 < 0.881 ✗ |
| "Bank Earnings Q1" (4 members, last article 5 days ago) | 0.68 | 0.73 + 0.05 + 0.064 = **0.844** | 0.68 < 0.844 ✗ |

Best match: "Fed Rate Decisions" (0.82). Runner-up that passed: none → margin check passes automatically. **Article is assigned to "Fed Rate Decisions".**

Now if the article were 0.83 to Thread A and 0.81 to Thread B (both passing threshold), the margin is only 0.02 < 0.03 → **ambiguous, left unmatched** for LLM grouping in Step 3. Assigning to the wrong thread would pollute that thread's centroid and cause a chain reaction of bad matches downstream.

### 3. Group leftovers with AI
Articles that didn't match any thread go to Gemini AI. The LLM receives each article's title, date, author, keywords, and a brief summary (first 2 sentences). This context helps the LLM make better grouping decisions — e.g., same author writing about the same topic across days likely belongs together.

The LLM groups articles and names each group (e.g., "Fed Holds Rates at 4.5%"). If the new group looks nearly identical to an existing thread (>0.92 similarity), we merge instead of creating a duplicate.

### 4. Retire old threads
Threads with no new articles for 14+ days are archived. They're still visible in article detail pages but won't accept new matches.

## Thread Lifecycle

```
Active → Cooling → Archived → (Resurrected → Active)
```

| State | Condition | Matching | Frontend |
|-------|-----------|----------|----------|
| **Active** | New article within 3 days | Yes (base threshold) | Shown, ranked by heat |
| **Cooling** | 3-14 days since last article | Yes (higher threshold) | Shown below active |
| **Archived** | 14+ days inactive | No matching | Hidden from feed, accessible via article detail timeline |
| **Resurrected** | Archived thread gets a new match | Becomes Active again | Re-appears in feed |

No hard time-based deletion. Threads persist indefinitely for historical timeline views.

**Implementation note**: Code uses a `status` TEXT column with values `'active'`, `'cooling'`, `'archived'`. Status transitions are managed by `update_thread_statuses()` based on `last_seen` date relative to `THREAD_COOLING_DAYS` (3) and `THREAD_ARCHIVE_DAYS` (14). Resurrection sets `status='active'` when a new article matches a cooling/archived thread.

## Heat Score

Computed at query time (not stored). Determines frontend ranking.

```
heat = Σ (importance_weight × time_decay(article_age))

importance_weight: must_read=3, worth_reading=2, optional=1
time_decay: e^(-0.3 × days_old)  → half-life ~2.3 days
```

Examples:
- "Fed Rate Decision" — today 2× must_read + yesterday 3× worth_reading → heat ≈ 10.5
- "Venezuela Crisis" — last article 5 days ago, 1× must_read → heat ≈ 0.67

## Daily Pipeline Flow

```
New articles crawled (RSS: title + description available immediately)
  → Embed with BAAI/bge-base-en-v1.5 (title + description + summary from LLM analysis)
  → Match against Active + Cooling thread centroids (dynamic threshold)
  → Unmatched articles → Gemini LLM grouping (title + date + author + summary + keywords)
  → Merge check: new thread centroid vs existing threads (>0.92 = merge)
  → Create remaining as new threads
  → Heat scores auto-update (query-time calculation)
```

### Embedding eligibility

All articles are eligible for embedding. Input is `title + description + summary` — the summary is joined from `wsj_llm_analysis` via `wsj_crawl_results`. If summary is not yet available (article not crawled/analyzed), `title + description` is used. The embedding model (bge-base-en-v1.5) has a 512-token limit; combined input fits within this for nearly all articles.

## Matching Rules

### Dynamic threshold (anti-gravity)

Each thread has its own dynamic threshold — the minimum cosine similarity an article needs to join. The threshold is higher for bigger/older threads, making them progressively harder to join.

```
effective_threshold = base(0.73) + time_penalty + size_penalty

time_penalty = 0.01 × days_gap       ← days since thread's last article
size_penalty = 0.04 × ln(member_count + 1)  ← logarithmic, so growth slows

Hard cap: if member_count >= 50 → effective_threshold = max(computed, 0.87)
  → Threads with 50+ members are near-frozen, only highly similar articles can join
Margin check: best_sim - second_best_sim must be >= 0.03, otherwise ambiguous → unmatched
  → Prevents coin-flip assignments that pollute centroids
```

| Thread size | size_penalty | effective (same day) | What it means |
|-------------|-------------|----------------------|---------------|
| 2 members | +0.044 | 0.774 | Easy to join — new thread still forming |
| 10 members | +0.096 | 0.826 | Moderate bar — established story |
| 50 members | +0.157 | **0.870** (hard cap) | Near-frozen — only clearly related articles |
| 100 members | +0.185 | **0.870** (hard cap) | Same as 50 — hard cap prevents further increase |

**Why anti-gravity matters:** Without size penalty, a big thread like "U.S. Economy" (high member count, broad centroid) would match loosely related articles at 0.74 similarity, absorb them, grow bigger, and attract even more — a gravity well. The size penalty breaks this cycle by raising the bar as threads grow.

**Note on base threshold:** 0.73 was tuned using Gemini-labeled ground truth (2,427 pairs, 333 articles). At the previous 0.62, contamination was 69% (39% "related" articles in same thread). At 0.73, contamination drops to 31% with 60% recall — missed articles are caught by LLM grouping in Step 3. See `docs/1.2.1-reranker-causal-test.md` for full analysis.

### Centroid update (Dynamic EMA)

When an article is assigned to a thread, the thread's centroid must be updated to include the new member. We use Exponential Moving Average (EMA) — a weighted blend of the old centroid and the new article's embedding.

The key: the blend weight (alpha) depends on thread size. Small threads let new articles shift the centroid significantly; large threads barely move.

```
effective_alpha = 0.1 / ln(member_count + 2)
new_centroid = effective_alpha × new_embedding + (1 - effective_alpha) × old_centroid
```

| Thread size | alpha | Effect | Example |
|-------------|-------|--------|---------|
| 2 members | 0.072 | New article = 7% influence | Thread direction still forming, new article can steer it |
| 10 members | 0.040 | 4% | Story is established, moderate influence |
| 50 members | 0.025 | 2.5% | Centroid is stable, hard to shift |
| 150 members | 0.020 | 2% — centroid barely moves | Even if a slightly off-topic article sneaks in, centroid stays put |

**Why this matters:** Without dynamic alpha, a single misclassified article about "general economy" joining a "Fed Rate" thread (50 members) could drag the centroid toward a broader meaning, making the thread match even more unrelated articles next time. With alpha=0.025, that one article shifts the centroid by only 2.5% — negligible. Meanwhile, a brand-new 2-member thread about "Tesla Recalls" can adapt quickly if the next article is "Tesla Recalls Cybertruck" (slightly different angle).

### LLM grouping

Articles that failed centroid matching (no thread was similar enough, or the match was ambiguous) are collected and sent to Gemini 2.5 Flash for semantic grouping.

**Why this step exists:** Centroid matching can only assign articles to *existing* threads. When a completely new story breaks (e.g., "Earthquake hits Tokyo"), there's no existing thread to match against. LLM grouping is the only way to create the initial thread for new topics.

**How it works:**
1. Collect unmatched articles.
2. Send them to Gemini with title, date, author, keywords, and summary (first 2 sentences, max 120 chars).
3. Gemini returns groups like: `["Earthquake hits Tokyo", "Japan earthquake death toll rises", "Tokyo infrastructure damage assessed"]` → Thread: "Tokyo Earthquake"
4. Each group is capped at 8 articles (larger groups are truncated to prevent overly broad threads).

**LLM input format per article:**
```
1. [a1b2c3d4] [2026-01-15] (Amrith Ramkumar) Fed Holds Rates at 4.5% [federal reserve, rates]
   > The Federal Reserve held its benchmark rate steady at 4.5%, signaling caution amid persistent inflation.
```

**Prompt rules include:** specific events only (no generic topics), 5-10 word titles naming entities/events/actions, max 8 articles per group, author as a grouping hint, prefer articles within 3 days of each other.

**No cross-validation:** Previously, LLM groups were verified via embedding cosine similarity (avg pairwise ≥ 0.60). This was removed because cosine similarity cannot capture causal relationships — e.g., "Fed holds rates" → "Housing responds" has low cosine but high causal relevance. The cross-validation was rejecting valid causal groups. Quality control will be revisited via golden dataset evaluation (Phase 4).

### Thread merge detection

After LLM grouping passes cross-validation, the pipeline computes a centroid for the new candidate thread and compares it against all existing Active/Cooling threads. If any existing thread is > **0.92** similar, the new articles are merged into that thread instead of creating a duplicate.

- **Why 0.92?** This is intentionally very high. Two threads about "Fed Rate Decision" from different days might be 0.93 similar → they merge (same ongoing story). But "Fed Chair Nomination" and "Trade Tariffs" might be 0.78 similar (both mention economic policy) → they stay separate. The high bar ensures only truly duplicate threads merge.
- The pipeline checks against the **best** match across all threads, not the first one found.

### Thread title refresh (NOT YET IMPLEMENTED)
Design: when a thread accumulates N+ new articles since last title update (thresholds: 10, 25, 50):
- Send last 5 article titles to Gemini
- Request updated thread title
- Keeps titles accurate as stories develop ("Fed Holds at 4.5%" → "Fed Cuts Rates to 4.25%")

**Status**: Not implemented in `7_embed_and_thread.py`. Thread titles are set at creation time by Gemini and never refreshed.

## Algorithm Constants

| Parameter | Value | Formula |
|-----------|-------|---------|
| Base threshold | 0.73 | Tuned via reranker experiment (docs/1.2.1) |
| Time penalty | 0.01/day | `0.01 × days_gap` |
| Size penalty | dynamic | `0.04 × ln(member_count + 1)` |
| Hard cap | 50 members | Threads ≥50 require 0.87+ similarity |
| Frozen threshold | 0.87 | Override for hard-capped threads |
| Match margin | 0.03 | Best must beat runner-up by this |
| EMA alpha | dynamic | `0.1 / ln(member_count + 2)` |
| Merge threshold | 0.92 | Fixed; finds best match, not first |
| LLM group max size | 8 | Max articles per LLM group |
| Archive after | 14 days | Days since last article |

## Edge Cases

### Thread quality — preventing contamination

<details>
<summary><strong>Gravity well</strong> — large thread absorbs everything</summary>

**Problem:** A 40-member "U.S. Economy" thread has a broad centroid. A new article about "Apple iPhone Sales" mentions economic impact → cosine similarity 0.74 → it joins. Thread gets broader, attracts more unrelated articles.

**Solution:** Size penalty (`0.04 × ln(members+1)`) raises the bar. At 40 members, threshold is 0.73 + 0.149 = **0.879**. The Apple article at 0.74 can't get in. At 50+ members, hard cap enforces 0.87 minimum.
</details>

<details>
<summary><strong>Centroid drift</strong> — thread shifts to "general news"</summary>

**Problem:** A "Fed Rate Decision" thread (50 members) gets 3 loosely related articles about "Bank Earnings" that pass the threshold. If each article shifts the centroid equally, after 3 articles the centroid drifts toward "general finance."

**Solution:** Dynamic EMA. At 50 members, alpha = 0.025 — each new article only shifts the centroid by 2.5%. Three articles shift it by ~7.5% total, which is not enough to meaningfully change what the thread matches next.
</details>

<details>
<summary><strong>Roundup pollution</strong> — Market Talk matches everything</summary>

**Problem:** "Roundup: Market Talk" articles summarize multiple sectors in one piece, creating embeddings that point toward "general market news" and match many threads.

**Solution:** Filtered at RSS ingest (`1_wsj_ingest.py`). These articles never enter the pipeline.
</details>

<details>
<summary><strong>LLM garbage groups</strong> — unrelated articles forced together</summary>

**Problem:** Gemini groups "China EV Tariffs" with "China Earthquake" because both mention China.

**Solution:** The LLM prompt includes strict rules against generic topic grouping and requires specific event/entity naming. The summary context helps the LLM distinguish between articles that share keywords but are about different events. Quality is further validated via golden dataset evaluation (Phase 4).
</details>

### Thread lifecycle — splitting, merging, and aging

<details>
<summary><strong>Near-duplicate threads</strong> — LLM creates a thread that already exists</summary>

**Problem:** An existing thread "Fed Holds Rates at 4.5%" exists. LLM groups new articles into "Federal Reserve Rate Decision" — essentially the same story.

**Solution:** Before creating the new thread, merge check compares centroids. If similarity > 0.92, the new articles merge into the existing thread instead.
</details>

<details>
<summary><strong>Same topic, different events</strong> — two separate shootings</summary>

**Problem:** "Texas School Shooting" thread exists (last article 10 days ago). A new "Ohio Mall Shooting" article has similarity 0.76 to it — similar topic but different event.

**Solution:** Time penalty adds +0.10 (10 days × 0.01). Threshold becomes 0.73 + 0.10 + size_penalty ≈ **0.87**. The 0.76 similarity doesn't pass → article goes to LLM grouping → creates a separate "Ohio Mall Shooting" thread.
</details>

<details>
<summary><strong>Story evolution</strong> — one story splits into sub-stories</summary>

**Problem:** "US-China Trade War" thread evolves. Some articles focus on "Agriculture Tariffs" while others focus on "Tech Export Bans." Should they stay in one thread or split?

**Solution:** Natural splitting. As the sub-stories diverge, their embeddings move apart. New "Agriculture Tariffs" articles may not pass the threshold for the broad "Trade War" thread. LLM grouping picks them up and creates a new focused thread.
</details>

<details>
<summary><strong>Long-running story</strong> — months-long saga</summary>

**Problem:** "Boeing 737 MAX Investigation" runs for 6 months. The story shifts from "crash cause" to "CEO fired" to "congressional hearings."

**Solution:** The centroid follows slowly (low alpha). If the shift is gradual, the thread adapts. If a dramatic shift happens (e.g., "Boeing stock fraud" — very different embedding), the new articles won't match the old centroid → LLM creates a new thread for the new angle.
</details>

<details>
<summary><strong>Resurrection</strong> — quiet story suddenly re-erupts</summary>

**Problem:** "Nord Stream Pipeline" thread has been archived for 30 days (no articles). A new investigation report drops.

**Solution:** Active and Cooling threads are in the matching pool. If a cooling thread gets matched, it is resurrected (`status='active'`, `last_seen` updated). Archived threads are NOT in the matching pool — if the same topic re-emerges, LLM grouping creates a new thread.
</details>

### Volume and assignment edge cases

<details>
<summary><strong>Volume spike</strong> — 50 articles/day during market crash</summary>

**Problem:** A market crash generates 50 related articles in one day. Will the system handle it?

**Solution:** Centroid matching scales linearly — each article is compared against thread centroids regardless of volume. The first few articles join existing threads or create new ones via LLM. Subsequent articles match the now-established threads. Anti-gravity prevents the "Market Crash" thread from becoming a black hole.
</details>

<details>
<summary><strong>Article fits 2 threads equally</strong></summary>

**Problem:** "Fed Rate Decision Impact on Housing Market" is 0.81 similar to "Fed Rate Decision" and 0.79 similar to "Housing Market Slowdown."

**Solution:** Margin = 0.81 - 0.79 = 0.02 < 0.03 → **ambiguous, left unmatched**. LLM grouping in Step 3 decides. If it had been 0.81 vs 0.75 (margin 0.06), it would be assigned to "Fed Rate Decision." Note: each article can only belong to one thread (1:1 FK constraint).
</details>

<details>
<summary><strong>New topic appears</strong> — no existing thread to match</summary>

**Problem:** "Asteroid Mining Company IPO" — completely new topic, no thread has a similar centroid.

**Solution:** The article fails centroid matching (no thread above threshold). It enters LLM grouping with other recent unmatched articles. If Gemini finds 2+ articles about asteroid mining, it creates a new thread. If the article is unique, it stays unthreaded as a standalone.
</details>

## Database Schema

`wsj_story_threads` table uses `status TEXT NOT NULL DEFAULT 'active'` with values `'active'`, `'cooling'`, `'archived'`. Indexed via `idx_wsj_story_threads_status`. Migration: `supabase/migrations/012_thread_status.sql`.

Heat score is computed at query time from joined article data, not stored.

## Backfill Strategy

Process existing articles chronologically (oldest → newest) to simulate natural pipeline:

```
Day 1 (oldest): LLM creates initial threads
Day 2: centroid match existing → LLM groups remaining → merge check
Day 3+: same pattern
```

Use `--backfill-by-date --start-date YYYY-MM-DD` flag. Deactivation is skipped
during backfill (historical dates would incorrectly archive everything).
