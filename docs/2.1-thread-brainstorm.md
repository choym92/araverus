# Thread System Brainstorm

Discussion notes on parent threads, thread detail page, and display strategy.
For current implementation details, see `docs/2.1-thread-stories.md`.

---

## 1. Parent Threads

### Why
Multiple threads can cover the same macro story from different angles. Example — Iran:
```
Parent: "Iran Crisis"
  ├── US-Iran Military Escalation (72 members, active)     — military
  ├── European Gas Prices Amid US-Iran (50 members, active) — energy/economy
  ├── Escalating Protests in Iran (13 members, cooling)     — domestic
  └── US-Iran Diplomatic Efforts (10 members, cooling)      — diplomacy
```
These should NOT be merged (different aspects), but users want a single view of "everything Iran."

### Creation Timing: After Children (Bottom-Up)

1. Can't predict which threads will emerge before articles arrive
2. Parent grouping requires observing existing children
3. Top-down pre-definition is just category tagging (feed_name already does this)

**Flow**:
- Initial: children created → LLM groups into parents (periodic pass)
- Ongoing: new thread created → try matching to existing parent (centroid or title similarity) → orphan if no match → next grouping pass picks it up

### Creation Method: LLM Direct Grouping

Active threads are ~30-50 at any time. Small enough for direct LLM grouping:
1. Get all active + cooling thread titles + member_count
2. Send to Gemini: "Group threads about the SAME macro event from different angles. Don't group by sector alone."
3. LLM returns groupings + parent titles
4. Create/update `wsj_parent_threads`, set `thread.parent_id`

Why not centroid-based: "Iran Military" and "European Gas Prices" have very different centroids (military vs energy embeddings). Only LLM can judge "same macro story."

### Schema Options

**Option A: Separate table** (current plan in `2.1-thread-stories.md`)
```sql
wsj_parent_threads (id, title, created_at)
wsj_story_threads.parent_id FK → wsj_parent_threads(id)
```

**Option B: Self-referencing FK**
```sql
wsj_story_threads.parent_thread_id FK → wsj_story_threads(id)
```
Simpler (no new table), but parent thread IS a regular thread with its own centroid/members. Semantically confusing — parents don't have articles directly.

**Leaning toward Option A** — cleaner separation.

## 2. Thread Size

### Current State
- Hard cap at 50 members → threshold jumps to 0.87
- But "US-Iran Military" has 72 members, "AI Transformation" has 54 — articles still matching at 0.87+
- This means the articles genuinely belong there

### Decision: Display Limit, Not Matching Limit

Don't cap matching — if articles match at 0.87+, they belong in the thread. Instead:
- Thread detail page: show **top 20 articles**, "Show all N" button
- Stories tab: **3-5 article** preview per thread
- ArticleCard carousel: already limited to visible slides

## 3. Thread Display Timeline

### Data Snapshot (2026-03-05)
- 31 active (last_seen within 3 days)
- 49 cooling (3-14 days)

### Display Strategy

| View | Scope | Sort |
|------|-------|------|
| Stories tab (default) | Active + Cooling (14 days) | Heat score |
| "Past Stories" section | Archived, last 30 days | member_count |
| Thread detail page | Always accessible | chronological |

All threads remain in DB forever. Display is the filter, not deletion.

## 4. Thread Detail Page (`/news/thread/[id]`)

Does not exist yet. This is the **landing page** for all other UI improvements.

### Content
- Thread title + summary (from `wsj_story_threads.title`)
