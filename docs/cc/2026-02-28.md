### Key Decisions & Rationale
| Decision | Reason |
|----------|--------|
| 4-phase overhaul instead of just re-embed | Mixed embeddings break threshold; status column + author signal are architectural improvements we want anyway |
| No backup tables (Phase 0 removed) | re-embed + re-thread can be re-run if needed (~35 min + ~$1-2 LLM cost). Backup tables add complexity, not needed for safety. |
| `status: TEXT` not ENUM | More flexible for future values (e.g. 'paused'). No migration cost. |
| 3-day cooling window | **NEEDS DISCUSSION** — currently a heuristic, not data-driven. Real question: what's the actual distribution of thread re-activation timing? |
| Author boost threshold 0.60 (lower than base 0.73) | Catches same-journalist-covering-story cases like Amrith Ramkumar on Anthropic/Pentagon where cosine drift caused split threads |
| 48-hour window for author boost | Arbitrary but reasonable; longer = more false positives, shorter = miss valid continuations |
| Keep `active` column during Phase 1, drop in Phase 5 | Safety: code changes first, then schema cleanup. Prevents runtime errors if deployment rolls back |

### Open Questions for Phase 1 Discussion

- **Status query performance**: Is the index on `status` enough, or should we denormalize cooling count to frontend cache?
- **Resurrection logic**: When archived thread gets new match, does it reset `last_seen` or keep old value for history?
