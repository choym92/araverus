# Thread Intelligence — Frontend

Thread tab (`/news?tab=threads`) displays story threads as portfolio-linked intelligence. For general `/news` architecture, see `docs/2-news-frontend.md`. For the threading algorithm and backend, see `docs/1.3-news-threading.md`. For the refactor roadmap (LLM judge + impact analysis), see `docs/workflow/4-todo/threading-refactor.md`.

## 1. Current Implementation

### Data Flow

```
page.tsx (Server Component)
  -> getStoriesData(category?) -- unstable_cache, 30-min TTL, tag: 'news'
    -> NewsService.getActiveThreadsGrouped(category?)
      -> Supabase: wsj_story_threads (active) + wsj_items (articles per thread)
      -> Calculate heat score per thread
      -> Group by parent_id (null = orphan group)
      -> Sort by totalHeat desc
  -> Pass parentThreadGroups to NewsContent
    -> NewsContent renders StoriesTab component
```

### Interfaces

```typescript
interface ParentThread {
  id: string;
  title: string;
}

interface StoryThreadWithDetails extends StoryThread {
  parent_id: string | null;
  heat: number;
  recentArticles: NewsItem[];
}

interface ParentThreadGroup {
  parent: ParentThread | null;  // null for orphan/ungrouped threads
  subThreads: StoryThreadWithDetails[];
  totalHeat: number;
}
```

### Heat Score Formula

```
heat = SUM(importance_weight * e^(-0.3 * days_old))

importance_weight:
  must_read     = 3
  worth_reading = 2
  optional      = 1

days_old = (now - article.created_at) / 86400
```

Parent totalHeat = sum of all sub-thread heats. Exponential decay (~26%/day) keeps recent threads on top while high-importance older threads still rank above new low-importance ones.

### Category Filtering

Server-side via `?category=` query parameter. Threads are cross-category — filtering shows only matching articles within each thread. Threads with zero matching articles are hidden.

### Tab Navigation

```typescript
const isDisabled = t.value === 'search';  // only Search tab is disabled
```

URL patterns:
- `/news?tab=threads` — all categories
- `/news?tab=threads&category=TECH` — Tech only

## 2. Parent Thread Grouping

### Why Parents

Multiple threads can cover the same macro story from different angles:
```
Parent: "Iran Crisis"
  |-- US-Iran Military Escalation (72 members, active)     -- military
  |-- European Gas Prices Amid US-Iran (50 members, active) -- energy/economy
  |-- Escalating Protests in Iran (13 members, cooling)     -- domestic
  |-- US-Iran Diplomatic Efforts (10 members, cooling)      -- diplomacy
```
These should NOT be merged (different aspects), but users want a single view of "everything Iran."

### Creation: Bottom-Up via LLM

Active threads are ~30-50 at any time. Small enough for direct LLM grouping:
1. Get all active + cooling thread titles + member_count
2. Send to Gemini: "Group threads about the SAME macro event from different angles. Don't group by sector alone."
3. LLM returns groupings + parent titles
4. Create/update `wsj_parent_threads`, set `thread.parent_id`

Why not centroid-based: "Iran Military" and "European Gas Prices" have very different centroids. Only LLM can judge "same macro story."

### Schema

```sql
wsj_parent_threads (id UUID PK, title TEXT, created_at TIMESTAMPTZ, status TEXT)
wsj_story_threads.parent_id FK -> wsj_parent_threads(id)  -- nullable
```

Separate table (not self-referencing FK) — cleaner separation since parents don't have articles directly.

### Thread Size: Display Limit, Not Matching Limit

Don't cap matching — if articles match at high similarity, they belong. Instead:
- Thread detail page: **top 20 articles**, "Show all N" button
- Threads tab: **3-5 article** preview per thread
- ArticleCard carousel: already limited to visible slides

### Display Timeline

| View | Scope | Sort |
|------|-------|------|
| Threads tab (default) | Active + Cooling (14 days) | Heat score |
| "Past Stories" section | Archived, last 30 days | member_count |
| Thread detail page | Always accessible | chronological |

All threads remain in DB forever. Display is the filter, not deletion.

## 3. Thread Impact Analysis (Post-Refactor)

> Depends on: `threading-refactor.md` Part 2. This section describes the frontend integration.

Each thread gets structured impact data stored in `wsj_story_threads.impacts` (JSONB):

```json
{
  "impacts": [
    {"type": "commodity", "name": "crude_oil", "confidence": "direct", "direction": "positive", "reason": "Hormuz strait disruption threatens oil supply"},
    {"type": "sector", "name": "airlines", "confidence": "indirect", "direction": "negative", "reason": "Oil price increase pressures fuel costs"}
  ]
}
```

### Frontend Display: Impact Chips

On parent cards and thread rows, show compact impact indicators:

```
crude oil ^  shipping v  defense ^  airlines v
[direct]     [direct]    [direct]   [indirect]
```

Styling:
- Direct impacts: solid background chip
- Indirect: outline/ghost chip
- Direction arrows: `^` positive, `v` negative, `~` volatile

### Thread Detail: Market Impact Section

Between sub-thread filter chips and article timeline:

```
Market Impact
  Direct Exposure
    crude oil ^    Hormuz strait disruption threatens supply
    shipping v     Maritime disruption risk in Persian Gulf
    defense ^      Military escalation drives demand narrative
  Indirect Exposure
    airlines v     Oil price increase pressures fuel costs
    insurance v    Shipping insurance premiums rise on conflict risk
```

## 4. Portfolio Relevance (Future — Post-Refactor Part 3)

> Depends on: `threading-refactor.md` Part 3, `user_watchlist` table.

### How it Works

```
User watchlist: [NVDA, XOM, DAL, JPM]
  -> Map tickers to sectors (ticker_sectors.py)
  -> Find threads with impacts matching those sectors
  -> Rank by: direct > indirect, recency, heat score
  -> Display: "Market Exposure" badges on relevant threads
```

### Frontend: Market Exposure Badge

On thread cards where user has relevant holdings:

```
Market Exposure: XOM (energy) | DAL (airlines)
```

**Wording**: "Market Exposure" not "Relevant to" — positions as financial intelligence tool, not news recommendation. Never "buy/sell" language.

### Portfolio Filter

Filter bar on Threads tab:

```
[All Threads] [My Portfolio] [Must Read]
```

"My Portfolio" shows only threads with impacts matching user's watchlist tickers.

## 7. Causal Thread Links (Future)

> With the LLM judge refactor, causal relationships are captured at thread assignment time. This section covers the **explicit link storage** for cross-thread causal display.

### Schema

```sql
CREATE TABLE wsj_thread_links (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source_thread_id UUID NOT NULL REFERENCES wsj_story_threads(id),
    target_thread_id UUID NOT NULL REFERENCES wsj_story_threads(id),
    relationship TEXT NOT NULL,        -- 'causes', 'reacts_to', 'escalates'
    confidence FLOAT,                  -- LLM confidence score
    evidence_summary TEXT,             -- LLM reasoning
    created_at TIMESTAMPTZ DEFAULT now(),
    UNIQUE(source_thread_id, target_thread_id)
);
```

### Frontend: "Related Storylines"

On article detail page, below Story Timeline:
```
Related Storylines
  This story is connected to:
  S&P 500 Rally (3 articles)
    "Fed rate cut drove market optimism"
  Housing Market Surge (2 articles)
    "Lower rates boosted mortgage demand"
```

### Many-to-Many (Deferred)

If causal links exist, articles could belong to multiple threads via `wsj_item_threads` junction table. Defer until Threads tab is mature.

## 8. Known Limitations & Open Questions

### Current Limitations

1. **Flat structure** — All threads have `parent_id = null` until backend migration adds `wsj_parent_threads` table
2. **Heat calculation in JS** — Acceptable for <=50 threads; move to SQL if >=100 become typical
3. **Category filter at article level** — Correct behavior (threads are cross-category), but can be confusing
4. **No impact data** — `impacts` column doesn't exist yet (threading-refactor.md Phase 2)
5. **No portfolio features** — `user_watchlist` table doesn't exist yet (Phase 3)
6. **1:1 FK constraint** — `wsj_items.thread_id` means one article belongs to one thread

### Open Questions

- **Thread quality monitoring**: Add avg pairwise similarity, member count distribution, creation/archive rate to `pipeline_health.py`
- **Thread title quality**: Are Gemini-generated titles good enough for user-facing display?
- **Category-level thread exposure**: Currently global heat ranking — no guarantee of "top 3 per category" coverage
- **Tab rename timing**: When to rename "Stories" -> "Threads" in UI (coordinate with backend refactor?)
- **Portfolio MVP**: localStorage watchlist (no auth) vs Supabase `user_watchlist` (requires auth)
- **Regulatory framing**: All copy must say "relevance"/"exposure" not "recommendation." Review SEC publisher's exclusion criteria before portfolio features launch.
