## Phase 6: Parent Thread Grouping (NOT YET IMPLEMENTED)

### Concept
- **Parent thread** = LLM-generated group title. Does not hold articles directly.
- **Thread (sub)** = Current thread. Holds articles, centroid, status.
- Multiple threads belong to one parent. Independent threads have `parent_id = NULL`.

### Example
```
Parent: "Anthropic vs Pentagon AI Battle" (LLM title)
  +-- Thread: "Pentagon Embraces OpenAI Over Anthropic" (3 articles)
  +-- Thread: "Anthropic Safety Guardrails Debate" (2 articles)
  +-- Thread: "What's at Stake for AI Industry" (1 article)

Parent: "AI Industry Workforce Impact"
  +-- Thread: "Block Cuts 4,000 Jobs" (4 articles)
  +-- Thread: "Tech Layoff Wave Concerns" (2 articles)

Thread: "Nvidia New Inference Chip" (3 articles, no parent)
```

### Migration: `supabase/migrations/013_parent_threads.sql`
```sql
CREATE TABLE wsj_parent_threads (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    title TEXT NOT NULL,
    created_at TIMESTAMPTZ DEFAULT now(),
    updated_at TIMESTAMPTZ DEFAULT now()
);

ALTER TABLE wsj_story_threads ADD COLUMN parent_id UUID REFERENCES wsj_parent_threads(id);
CREATE INDEX idx_wsj_story_threads_parent ON wsj_story_threads(parent_id);
```

### Grouping Logic
- **When**: After daily threading creates all threads, run parent grouping as a batch.
- **How**: LLM takes all active/cooling thread titles + centroids -> groups related -> generates parent title.
- **Threshold**: Thread centroid similarity > TBD (likely 0.70-0.80) for same parent candidates.
- **Daily**: When new thread is created, check if it belongs to an existing parent.

### Code Changes
- `7_embed_and_thread.py`: New step — `group_threads_into_parents()` after thread assignment
- `news-service.ts`: `ParentThread` interface, query threads grouped by parent
- Frontend: Display parent as collapsible group header

### Decisions
- Parent centroid: YES — avg of child centroids
- Parent title update: NO — keep initial LLM title
- Parent status: NONE — derived from child statuses
- Drift auto-detach: DEFERRED — observe 2-3 weeks first
- Frontend UX: SEPARATE — handled in frontend plan (StoriesTab)

### Rollback
```sql
ALTER TABLE wsj_story_threads DROP COLUMN parent_id;
DROP TABLE wsj_parent_threads;
```

# 2. Cross-Encoder Reranker Experiment

> Origin: `docs/1.2.1-reranker-causal-test.md`
> Notebook: `notebooks/reranker_causal_test.ipynb`
> Data: `notebooks/reranker_causal_labels.pkl` (2,427 labeled pairs, 333 articles)

## Overview

Tested whether cross-encoder rerankers can improve threading beyond cosine similarity. Two tasks: same_story detection and causal relationship detection.

## Two Tasks

| Task | Question | Cosine capability |
|------|----------|--------------------|
| **same_story** | Same developing story? | Yes — centroid matching |
| **causal** | Does A's event cause B's event? | No — cosine measures similarity, not causation |

## Experiment Design

### Data Collection
- **Source**: 333 articles from last 7 days
- **Candidate pairs**: Entity/ticker overlap (790) + Embedding neighbors (1,437) + Random negatives (200) = **2,427 pairs**
- **Rules**: time-ordered, <=3 days apart, deduplicated

### Ground Truth Labeling
- **Model**: Gemini 2.5 Flash (thinking model)
- **Approach**: 175 batches, ~14 pairs/batch, ~103 minutes
- **Labels**: same_story (same/related/unrelated), causal (A_causes_B/B_causes_A/no/unclear)
- **Quality**: 2,419/2,427 labeled, 0 temporal violations, spot-check accurate

### Label Distribution

| same_story | count | causal | count |
|------------|-------|--------|-------|
| unrelated | 1,150 | no | 1,943 |
| related | 975 | A_causes_B | 291 |
| same | 302 | unclear | 160 |
| | | B_causes_A | 33 |

## Models Tested

| Model | Type | Size |
|-------|------|------|
| BAAI/bge-base-en-v1.5 | Bi-encoder (baseline) | 109M |
| BAAI/bge-reranker-v2-m3 | Cross-encoder | 568M |
| Qwen3-Reranker-0.6B | CausalLM | 600M |
| tomaarsen/Qwen3-Reranker-0.6B-seq-cls | CrossEncoder (seq-cls) | 600M |

## Full Results

| Model | same_story AUC | same_story F1 | causal AUC | causal F1 | Runtime |
|-------|---------------|--------------|-----------|----------|---------|
| bge-base cosine (baseline) | 0.8728 | 0.6198 | 0.8081 | 0.4703 | <1s |
| bge-reranker-v2-m3 | collapsed | — | collapsed | — | 53s |
| Qwen3-Reranker (plain) | 0.9291 | 0.7465 | 0.8551 | 0.5014 | 229s |
| **Qwen3-Reranker (enriched)** | **0.9425** | **0.7640** | **0.8695** | **0.5181** | 822s |
| Qwen3-seq-cls (plain) | 0.7429 | — | 0.6815 | — | ~60s |
| Qwen3-seq-cls (enriched) | 0.7511 | — | 0.6902 | — | ~60s |

## Critical Analysis

1. Random negatives inflate AUC — any model separates unrelated pairs
2. Qwen3 binary-like scores make threshold selection brittle
3. Circular labels: Gemini labels may share Qwen3's LLM biases
4. Weak causal F1 (0.52) means low real-world precision
5. 822s runtime impractical for daily pipeline
6. Enriched adds only +1.3pp AUC for 4x runtime

**Verdict: No-Go for Qwen3 as cosine replacement.**

## Threshold Tuning (from same experiment)

| Threshold | Predicted | Precision | Recall | Contamination | F1 |
|-----------|-----------|-----------|--------|---------------|----|
| **0.62** (was) | 886 | 0.306 | 0.898 | **0.694** | 0.456 |
| **0.70** (best F1) | 404 | 0.540 | 0.722 | **0.460** | 0.617 |
| **0.73** (applied) | 261 | 0.690 | 0.596 | **0.310** | 0.640 |

Applied: `THREAD_BASE_THRESHOLD` 0.62 -> 0.73, `LLM_GROUP_MIN_SIMILARITY` 0.42 -> 0.60.

## Phase 2: Re-evaluation After Grid Search

Grid search revealed **47.8% fragmentation** — threshold tuning alone can't solve it. This contradicted the original No-Go conclusion.

**Updated verdict**: Cross-encoder is **Go as complement**, not replacement:
- Cosine for fast filtering (>0.75 match, <0.40 reject)
- CE for ambiguous zone (0.40-0.75)

## Phase 3: Golden v2.1 + Alternative Approaches

### Golden Dataset v2.1
v1 golden (134 threads) had garbage-bin threads. Rebuilt via split-then-merge:
1. HDBSCAN split large threads
2. Gemini validated sub-clusters
3. Gemini merge pass recombined over-split

# 3. Grid Search Report v1

> Origin: `docs/4-threading-gridsearch-report.md`
> Korean translation: available in git history (`docs/4-threading-gridsearch-report-ko.md`)
> Note: This report used golden v1 (134 threads). Superseded by v2.1 grid search in Section 2 Phase 3.

## Golden Dataset v1 Creation

4-step process:

1. **Over-thread with low thresholds** (base=0.58): 943 threads from 2,102 articles, 167 with 2+ articles
2. **Gemini Q1 — Thread validation**: 39/167 coherent, 128 need splitting
3. **Gemini Q2 — Thread pair comparison**: 6 merges, 2 parent-child, 42 separate
4. **Gemini Q3 — Singleton review**: 86 new groups, 19 join existing, 527 truly standalone

**Final**: 134 threads, 913 articles, 556 singletons, 61 inter-article links (25 causal)

## Evaluation Metrics

| Metric | Formula | Weight |
|--------|---------|--------|
| **Contamination** | contaminated threads / total threads | 0.3 |
| **Fragmentation** | fragmented golden threads / total golden | 0.3 |
| **Causal Recall** | recalled causal links / total causal links | 0.4 |

Composite = 0.3*(1-contamination) + 0.3*(1-fragmentation) + 0.4*causal_recall

## Grid Search Design

5 parameters, 1,536 combinations:
- `base_threshold`: [0.60, 0.65, 0.68, 0.70, 0.73, 0.75, 0.78, 0.80]
- `author_threshold`: [0.50, 0.55, 0.60, 0.65]
- `time_penalty`: [0.005, 0.01, 0.015, 0.02]
- `centroid_decay`: [0.05, 0.08, 0.10, 0.15]
- `entity_weight`: [0.02, 0.04, 0.06]

All in-memory numpy — no DB or LLM calls. Optimized from ~7h to ~40 min (11x speedup).

## Results

| Metric | Production (rank 693/1536) | Best (rank 1/1536) |
|--------|--------------------------|---------------------|
| **Composite** | 0.6867 | **0.8076** (+17.6%) |
| Contamination | 0.1348 | **0.1105** |
| Fragmentation | 0.5896 | **0.4776** |
| Causal Recall | 0.7600 | **0.9600** |

### Parameter Sensitivity

| Parameter | Best | Sensitivity |
|-----------|------|-------------|
| `base_threshold` | **0.65** | **HIGH** (range: 0.192) |
| `time_penalty` | **0.005** | MEDIUM (range: 0.079) |
| `author_threshold` | **0.60** | LOW |
| `centroid_decay` | **0.05** | NONE |
| `entity_weight` | **0.06** | NONE |

**base_threshold is the only parameter that meaningfully matters.**

## Critical Analysis

1. **Circular reasoning**: Golden dataset created from low-threshold simulation + Gemini validation — inherently rewards aggressive threading
2. **Simulation != production**: 84.3% mismatch (simulator: 225 threads, DB: 102). Simulator lacks LLM grouping
3. **Only 1 param matters**: centroid_decay has zero sensitivity — evaluation may be too coarse
4. **Causal recall fragile**: Only 25 links. Production vs best difference = 5 links
5. **Fragmentation ceiling**: 47.8% even at best params — fundamental cosine limitation

## Recommended Params (v1 — later superseded by v2.1)

```python
THREAD_BASE_THRESHOLD = 0.65      # was 0.73
THREAD_TIME_PENALTY = 0.005       # was 0.01
CENTROID_DECAY = 0.08             # was 0.10
ENTITY_WEIGHT = 0.02              # was 0.04
```

## Future: Causal Thread Links

Keep current topical threading as-is. Add a separate causal relationship layer on top.

```
Topical threads (existing):
  [Fed Rate Policy]  ****
  [Stock Market]     ***
  [Housing Market]   **

Causal links (new layer):
  [Fed Rate Policy] --causes--> [Stock Market]
  [Fed Rate Policy] --causes--> [Housing Market]
  [Stock Market]    --causes--> [Housing Market]
```

### Schema
```sql
CREATE TABLE wsj_thread_links (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source_thread_id UUID NOT NULL REFERENCES wsj_story_threads(id),
    target_thread_id UUID NOT NULL REFERENCES wsj_story_threads(id),
    relationship TEXT NOT NULL,        -- 'causes', 'reacts_to', 'escalates'
    confidence FLOAT,
    evidence_summary TEXT,
    created_at TIMESTAMPTZ DEFAULT now(),
    UNIQUE(source_thread_id, target_thread_id)
);
```

### Detection Pipeline
1. After daily threading, collect active threads + recent article titles
2. Send to Gemini: "Which threads have causal relationships?"
3. Store in `wsj_thread_links`
4. Purely LLM judgment — no embedding validation needed

### Frontend: "Related Storylines"
On article detail page, below Story Timeline:
```
+-- Related Storylines ---------------------+
|  This story is connected to:              |
|                                           |
|  S&P 500 Rally (3 articles)              |
|     "Fed rate cut drove market optimism"  |
|                                           |
|  Housing Market Surge (2 articles)        |
|     "Lower rates boosted mortgage demand" |
+-------------------------------------------+
```
